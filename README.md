# Supervised App-to-CLI Orchestration Template

Korean user guide: [README.ko.md](README.ko.md)

## What this repository is for

This repository is a bootstrap and operating foundation for future projects.
It is now set up as a supervised orchestration template where:

- Codex app is the human-facing supervisor and approval surface.
- Codex CLI is the worker engine for read-only fan-out and approved execution.
- Planning starts by inspecting a repo-managed skill catalog and routing rules before task decomposition.
- CLI runs write artifacts into the repository so the app can read and summarize them.
- Product/runtime workflows should remain Docker-first when a real product stack is later chosen.

## Current status

Foundation only.

## Prerequisites

- Windows 11 host
- Git
- Docker Desktop with `docker compose`
- Codex desktop app in Local mode
- Codex CLI installed on the host and authenticated

The orchestration layer is host-local because Codex app and Codex CLI are host tools.
That does not change the Docker-first principle for future product build/test/runtime work.

## Quick start

1. Validate Docker compose if needed:

   ```powershell
   docker compose config
   ```

2. Check that Codex CLI is available:

   ```powershell
   codex --help
   ```

3. Run a supervised planning pass:

   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File scripts\orchestration\Invoke-SupervisorPlan.ps1 -Task "Describe the task here"
   ```

4. Read the latest run artifacts under `.codex/runs/<run-id>/`, including the skill-routing summary in `result.md` and `result.json`.

5. After human approval, run the execution handoff against an approved plan run:

   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File scripts\orchestration\Invoke-SupervisorExecute.ps1 -PlanRun <run-id> -ApprovalNote "Approved scope: ..."
   ```

6. Run a read-only review pass when needed:

   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File scripts\orchestration\Invoke-SupervisorReview.ps1 -TargetRun <run-id>
   ```

## Supervisor-first workflow

Default workflow:

1. Human asks in Codex app.
2. App uses a repo-local skill or repo action to invoke a bridge script.
3. The bridge script injects the skill catalog and routing rules, then runs `codex exec` with repo-local profiles and schemas.
4. Codex CLI writes structured output and logs into `.codex/runs/<run-id>/`.
5. Codex app reads `result.md` and `result.json` and summarizes them, including skill-routing decisions.
6. If the next step is mutating, the app asks the human for approval before execution.
7. Approved execution happens in an isolated git worktree.

Important limitation:
- The app is not treated as a live viewer of CLI child-agent internals. Artifact files are the bridge.

## Key paths

- `.codex/config.toml`: repo-local Codex config, profiles, and role definitions
- `.codex/agents/`: role-specific config files
- `.codex/instructions/`: root and role instruction files
- `.codex/skill-routing/`: repo-managed skill catalog and deterministic routing rules
- `.codex/schemas/`: JSON Schemas for plan, execution, and review outputs
- `.codex/runs/`: ignored run artifacts generated by the bridge scripts
- `scripts/orchestration/`: PowerShell bridge scripts
- `.agents/skills/`: repo-local Codex app skills

Human approval is required before:

- expanding scope beyond the approved slice
- destructive actions
- adding heavy dependencies or infrastructure
- turning this repo into a concrete product repo

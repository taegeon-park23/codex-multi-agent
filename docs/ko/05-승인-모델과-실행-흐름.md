# 05. 승인 모델과 실행 흐름

[인덱스](README.md) | [이전: 04. 코덱스 앱 사용 흐름](04-%EC%BD%94%EB%8D%B1%EC%8A%A4-%EC%95%B1-%EC%82%AC%EC%9A%A9-%ED%9D%90%EB%A6%84.md) | [다음: 06. 복붙용 프롬프트 모음](06-%EB%B3%B5%EB%B6%99%EC%9A%A9-%ED%94%84%EB%A1%AC%ED%94%84%ED%8A%B8-%EB%AA%A8%EC%9D%8C.md)

이 저장소의 가장 중요한 규칙은 "먼저 읽기 전용 계획, 그다음 사람 승인, 그다음 작은 실행"입니다.

## 기본 원칙

- 기본은 읽기 전용 계획과 리뷰입니다.
- 실행은 반드시 사전 승인 후에만 합니다.
- 실행은 기본적으로 격리된 worktree에서 진행합니다.

## 왜 이렇게 엄격한가요?

이 규칙은 단순히 보수적이어서가 아니라 도구의 실제 한계와 안전성 때문입니다.

핵심 이유:

- Codex CLI는 실제 작업을 돌립니다.
- 하지만 그 내부의 모든 승인 과정을 Codex app에서 항상 깔끔하게 보여 줄 수 있다고 가정하면 안 됩니다.
- 그래서 이 저장소는 "숨겨진 자동 승인"을 허용하지 않고, 사람이 먼저 명확하게 승인한 뒤에만 실행으로 넘어가도록 설계돼 있습니다.

## 기본 흐름

1. 읽기 전용 계획을 만든다.
2. 결과를 사람이 읽는다.
3. 실행이 필요하면 사람의 명시적 승인을 받는다.
4. 승인된 범위만 worktree에서 실행한다.
5. 실행 결과를 다시 리뷰한다.

## 무엇을 승인해야 하나요?

아래처럼 구체적이고 작은 범위만 승인해야 합니다.

- 문서 초안 정리 1건
- 작업 분해 결과 중 1개 작은 단위
- 특정 버그 1건 수정
- 아키텍처 초안 중 한 부분만 보완

## 무엇은 승인하면 안 되나요?

다음 상황에서는 승인하지 않는 편이 안전합니다.

- 범위가 너무 커서 한 줄로 설명되지 않을 때
- 여러 기능을 한꺼번에 넣으려 할 때
- 무거운 의존성 추가가 숨어 있을 때
- 인프라 변경이 포함되는데 아직 검토하지 않았을 때
- 결과 아티팩트를 아직 읽지 않았을 때
- 실행이 worktree 밖에서 이뤄질 가능성이 있을 때

## 승인 체크리스트

승인 전에 아래를 확인하세요.

- 내가 요청한 범위와 같은가
- 범위가 충분히 작은가
- 무거운 의존성이나 인프라 추가가 숨어 있지 않은가
- worktree 격리가 유지되는가
- 남은 위험과 열린 질문이 정리돼 있는가

## 실행 단계에서 기억할 점

- 실행은 계획의 연장선입니다.
- 실행은 새로 범위를 넓히는 단계가 아닙니다.
- 승인된 범위를 벗어나면 멈추고 다시 계획 또는 재승인으로 돌아가야 합니다.

## 좋은 승인 예시

```text
승인합니다. 범위는 PRD 초안과 요구사항 초안 정리까지로 제한해 주세요. 제품 스택 선택, 무거운 의존성 추가, 인프라 설정은 이번 실행 범위에 포함하지 마세요.
```

## 좋지 않은 승인 예시

```text
알아서 전부 진행해 주세요.
```

이런 식의 승인은 범위가 너무 넓고, 이 저장소의 승인 모델과 맞지 않습니다.

## 승인 후에도 확인해야 할 것

- 실행 결과가 승인한 범위 안에 머물렀는가
- 추가 승인이 필요한 이유가 생겼는가
- 다음 단계가 다시 읽기 전용 리뷰로 돌아가는가

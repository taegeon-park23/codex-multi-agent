# 03. 새 프로젝트 시작하기

[인덱스](README.md) | [이전: 02. 시작 전 준비](02-%EC%8B%9C%EC%9E%91-%EC%A0%84-%EC%A4%80%EB%B9%84.md) | [다음: 04. 코덱스 앱 사용 흐름](04-%EC%BD%94%EB%8D%B1%EC%8A%A4-%EC%95%B1-%EC%82%AC%EC%9A%A9-%ED%9D%90%EB%A6%84.md)

이 문서는 이 템플릿을 포크해서 실제 새 프로젝트를 시작하는 가장 안전한 기본 흐름을 설명합니다.

## 가장 안전한 시작 순서

1. 이 저장소를 포크한다.
2. 자신의 포크를 클론한다.
3. 새 프로젝트 저장소 위치로 정리한다.
4. Codex app에서 연다.
5. Docker와 Codex CLI를 확인한다.
6. 만들고 싶은 프로그램을 자세히 적는다.
7. 먼저 읽기 전용 계획 실행을 한다.
8. 결과 아티팩트를 읽는다.
9. 아주 작은 실행 범위만 승인한다.
10. 반복한다.

## 단계별 설명

### 1. 이 저장소를 포크한다

실제 프로젝트에 쓸 때는 원본 저장소를 직접 바꾸기보다 자신의 GitHub 계정으로 포크하는 것이 좋습니다.

이유:

- 자신의 프로젝트 기록을 분리할 수 있습니다.
- 실험과 변경을 안전하게 관리할 수 있습니다.
- 이후에는 이 저장소가 "내 프로젝트 저장소" 역할을 하게 됩니다.

### 2. 자신의 포크를 클론한다

예시:

```powershell
git clone https://github.com/<your-account>/<your-project-repo>.git
cd <your-project-repo>
```

### 3. 새 프로젝트 저장소 위치로 정리한다

다음 중 하나로 정리하면 됩니다.

- GitHub에서 저장소 이름을 프로젝트 이름에 맞게 바꾼다
- 프로젝트용 폴더 아래에 둔다
- 팀에서 정한 개발 루트 폴더 아래에 둔다
- Codex app에서 바로 열기 쉬운 위치에 둔다

중요한 것은 이름보다도 "이제부터 이 저장소가 내 프로젝트의 운영 기반"이라는 점입니다.

### 4. Codex app에서 연다

- 저장소 루트를 워크스페이스로 연다
- Local mode를 사용한다
- 첫 요청은 구현이 아니라 계획 요청으로 시작한다

### 5. Docker와 Codex CLI를 확인한다

```powershell
docker compose config
codex --help
codex exec --help
```

### 6. 만들고 싶은 프로그램을 자세히 적는다

좋은 첫 요청은 기술 이름보다 제품 의도를 잘 설명합니다.

아래처럼 적는 것이 좋습니다.

- 누구를 위한 프로그램인지
- 어떤 문제를 해결하는지
- 가장 중요한 기능 3~5개가 무엇인지
- 지금 아는 제약이 무엇인지
- 아직 결정하지 않은 기술 선택이 무엇인지

### 7. 먼저 읽기 전용 계획 실행을 한다

첫 실행은 구현이 아니라 계획이어야 합니다.

예시 명령:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\orchestration\Invoke-SupervisorPlan.ps1 -Task "여기에 만들고 싶은 프로그램 설명"
```

또는 Codex app에서 한국어 프롬프트를 넣고, 앱이 repo-local workflow를 통해 계획 스크립트를 호출하도록 사용할 수 있습니다.

### 8. 결과 아티팩트를 읽는다

확인 위치:

- `.codex/runs/<run-id>/request.md`
- `.codex/runs/<run-id>/effective-prompt.md`
- `.codex/runs/<run-id>/result.json`
- `.codex/runs/<run-id>/result.md`
- 필요하면 `.codex/runs/<run-id>/logs/stderr.log`

### 9. 아주 작은 실행 범위만 승인한다

처음부터 큰 범위를 한 번에 승인하지 마세요.

다음처럼 작은 범위를 고릅니다.

- 문서 초안 1개 정리
- 요구사항을 작은 작업 단위로 분해
- 한 가지 핵심 기능의 뼈대만 구현
- 테스트 가능한 작은 버그 하나만 수정

### 10. 반복한다

안전한 반복 순서:

1. 계획
2. 검토
3. 작은 범위 승인
4. 실행
5. 리뷰
6. 다음 계획

## 프로젝트 설명 예시

### 웹 앱 예시

```text
소규모 팀이 사용하는 내부 일정 관리 웹 앱을 만들고 싶습니다. 사용자는 팀 리더와 팀원입니다. 핵심 기능은 일정 생성, 담당자 배정, 진행 상태 확인, 간단한 알림입니다. 아직 기술 스택은 정하지 않았고, 먼저 요구사항과 정보 구조를 정리하고 싶습니다.
```

### 채팅 시스템 예시

```text
커뮤니티 운영자가 관리할 수 있는 실시간 채팅 시스템을 만들고 싶습니다. 공개 채널과 관리자 채널이 필요하고, 신고 처리와 대화 로그 보존 규칙을 먼저 정리하고 싶습니다. 아직 제품 구조와 저장 전략은 미정입니다.
```

### 시뮬레이션 예시

```text
창고 물류 흐름을 실험할 수 있는 시뮬레이션 도구를 만들고 싶습니다. 입고, 출고, 작업자 이동, 병목 구간을 모델링하고 싶고, 먼저 핵심 객체와 시나리오를 정의하고 싶습니다.
```

### 게임 예시

```text
짧은 세션으로 플레이하는 2D 전략 퍼즐 게임을 만들고 싶습니다. 자원, 턴, 스테이지 목표, 실패 조건을 먼저 정리하고 싶습니다. 아직 엔진이나 구현 방식은 정하지 않았습니다.
```

### 데스크톱 도구 예시

```text
반복 파일 작업을 줄여 주는 데스크톱 도구를 만들고 싶습니다. 폴더 비교, 이름 변경 규칙 적용, 작업 미리보기, 실행 기록 저장이 필요합니다. 먼저 사용자 흐름과 최소 기능 범위를 정리하고 싶습니다.
```

## 큰 범위를 한 번에 승인하지 않는 체크리스트

- 이번 실행이 한 문장으로 설명될 만큼 작은가
- 결과를 내가 직접 검토할 수 있는가
- 실패했을 때 되돌리기 쉬운가
- 무거운 의존성 추가가 숨어 있지 않은가
- 다음 단계로 넘어가기 전에 리뷰가 가능한가

## 추천

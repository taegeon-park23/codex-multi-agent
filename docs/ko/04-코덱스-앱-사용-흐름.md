# 04. 코덱스 앱 사용 흐름

[인덱스](README.md) | [이전: 03. 새 프로젝트 시작하기](03-%EC%83%88-%ED%94%84%EB%A1%9C%EC%A0%9D%ED%8A%B8-%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0.md) | [다음: 05. 승인 모델과 실행 흐름](05-%EC%8A%B9%EC%9D%B8-%EB%AA%A8%EB%8D%B8%EA%B3%BC-%EC%8B%A4%ED%96%89-%ED%9D%90%EB%A6%84.md)

이 저장소에서 사용자는 주로 Codex app를 통해 작업합니다.

하지만 실제 작업을 돌리는 엔진은 Codex CLI이고, 두 도구 사이를 이어 주는 것은 저장소 안의 PowerShell 브리지 스크립트입니다.

## 한눈에 보는 흐름

1. 사용자가 Codex app에 요청을 입력합니다.
2. 앱이 repo-local skill 또는 로컬 액션으로 스크립트를 호출합니다.
3. 스크립트가 Codex CLI를 실행합니다.
4. 결과가 `.codex/runs/<run-id>/` 아래에 저장됩니다.
5. 앱이 `result.md`와 `result.json`을 읽어 요약합니다.
6. 사람이 승인 여부를 결정합니다.

## 실제로는 어떻게 움직이나요?

### 1. 사용자가 앱에 요청을 입력한다

예:

- 새 프로젝트를 정의하고 싶다
- 현재 계획을 리뷰하고 싶다
- 승인한 계획 범위만 실행하고 싶다

### 2. 앱이 로컬 워크플로우를 호출한다

앱은 저장소 안의 워크플로우를 직접 실행하거나, repo-local skill의 안내에 따라 아래 스크립트 중 하나를 호출합니다.

- 계획: `scripts\orchestration\Invoke-SupervisorPlan.ps1`
- 실행: `scripts\orchestration\Invoke-SupervisorExecute.ps1`
- 리뷰: `scripts\orchestration\Invoke-SupervisorReview.ps1`

### 3. 스크립트가 Codex CLI를 실행한다

이 단계에서 실제 worker run이 시작됩니다.

중요한 점:

- 사용자는 여전히 Codex app에서 보고 판단합니다.
- 실제 내부 작업은 Codex CLI가 수행합니다.

### 4. 결과가 run artifacts로 저장된다

각 실행은 `.codex/runs/<run-id>/` 폴더에 기록됩니다.

사용자가 특히 확인해야 할 파일:

- `request.md`: 처음 요청이 무엇이었는지
- `effective-prompt.md`: 실제로 worker에 전달된 문맥
- `result.json`: 기계가 읽기 좋은 구조화 결과
- `result.md`: 사람이 빠르게 읽기 좋은 요약 결과
- `approval-request.md`: 추가 승인이 필요한 이유
- `logs/stderr.log`: 실패나 경고를 확인할 때 보는 로그

## 사용자가 무엇을 확인해야 하나요?

### 계획 단계에서

- 요청한 내용이 맞게 해석되었는가
- 다음 단계 추천이 과도하지 않은가
- 승인 필요 여부가 분명한가
- 위험과 열린 질문이 정리되어 있는가

### 실행 단계 전에

- 정말 내가 승인한 범위만 실행하는가
- 실행이 작은 범위로 제한되어 있는가
- worktree 격리가 유지되는가

### 리뷰 단계에서

- 실제 결과가 기대와 맞는가
- 남은 위험이 무엇인가
- 다음으로 무엇을 승인해야 하는가

## Codex app이 직접 볼 수 없는 것

이 부분을 오해하면 안 됩니다.

Codex app은 Codex CLI 내부에서 일어나는 모든 하위 작업을 실시간으로 직접 보여 주는 도구로 가정하면 안 됩니다.

즉, 앱에서 바로 다 보인다고 기대하면 안 되는 것:

- 내부 child-agent 활동 전체
- 중간 상태 전체
- 숨겨진 내부 reasoning 흐름 전체

## 아티팩트로 확인해야 하는 것

다음 정보는 run artifacts를 기준으로 봐야 합니다.

- 실제 요청 내용
- 실제 전달 프롬프트
- 구조화된 결과
- 승인 필요 여부
- 위험, 열린 질문, 다음 단계
- 실패 원인과 오류 로그

## 가장 안전한 사용 습관

1. 계획부터 시작한다.
2. `result.md`와 `result.json`을 둘 다 본다.
3. 애매하면 `logs/stderr.log`까지 본다.
4. 승인 전에는 실행하지 않는다.
5. 승인하더라도 아주 작은 범위만 실행한다.

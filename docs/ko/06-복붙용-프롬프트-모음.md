# 06. 복붙용 프롬프트 모음

[인덱스](README.md) | [이전: 05. 승인 모델과 실행 흐름](05-%EC%8A%B9%EC%9D%B8-%EB%AA%A8%EB%8D%B8%EA%B3%BC-%EC%8B%A4%ED%96%89-%ED%9D%90%EB%A6%84.md) | [다음: 07. 자주 하는 실수와 해결법](07-%EC%9E%90%EC%A3%BC-%ED%95%98%EB%8A%94-%EC%8B%A4%EC%88%98%EC%99%80-%ED%95%B4%EA%B2%B0%EB%B2%95.md)

이 문서는 Codex app에서 바로 붙여 넣어 쓸 수 있는 한국어 프롬프트 예시 모음입니다.

각 프롬프트는 이 저장소의 현재 워크플로우에 맞춰 "계획 먼저, 승인 후 실행" 원칙을 따릅니다.

## 1. 첫 프로젝트 정의 프롬프트

언제 쓰는지:
- 이 템플릿을 포크한 뒤, 새 프로젝트를 처음 정의할 때

채워 넣을 입력:
- `[프로그램 설명]`

기대 결과:
- PRD 초안
- 요구사항 초안
- 아키텍처 초안
- 첫 번째로 승인할 만한 작은 실행 단위 제안

```text
이 저장소를 내 새 프로젝트의 운영 템플릿으로 사용하겠습니다. 아직 구현하지 말고, 먼저 읽기 전용 계획 단계로 진행해 주세요. 내가 만들 프로그램 설명을 바탕으로 PRD 초안, 요구사항 초안, 아키텍처 초안, 첫 번째로 승인할 만한 작은 실행 단위를 제안하고, 결과 아티팩트를 읽기 쉽게 요약해 주세요. 프로그램 설명: [프로그램 설명]
```

## 2. 계획 전용 프롬프트

언제 쓰는지:
- 구현 없이 계획과 위험 분석만 받고 싶을 때

채워 넣을 입력:
- `[작업 설명]`

기대 결과:
- 읽기 전용 계획
- 열린 질문
- 위험 요약

```text
이 작업은 아직 구현하지 말고 읽기 전용 계획만 진행해 주세요. 작업 설명: [작업 설명]. 목표, 열린 질문, 위험, 작업 분해, 추천 다음 단계를 정리하고 결과 아티팩트를 요약해 주세요.
```

## 3. 스택 검토 프롬프트

언제 쓰는지:
- 제품 스택이 아직 정해지지 않았고 후보를 비교하고 싶을 때

채워 넣을 입력:
- `[제품 설명]`
- `[제약 조건]`

기대 결과:
- 2~3개 현실적 후보
- 장단점
- Docker 관점 비교

```text
아직 기술 스택은 정하지 않았습니다. 읽기 전용 계획 단계에서 [제품 설명]과 [제약 조건]을 기준으로 현실적인 스택 후보 2~3개를 비교해 주세요. 각 후보의 의존성 부담, Docker 관점, 유지보수성, 이 저장소의 감독형 워크플로우와의 궁합을 설명하고 추천안을 제시해 주세요.
```

## 4. 작업 분해 프롬프트

언제 쓰는지:
- 큰 목표를 승인 가능한 작은 단위로 쪼개고 싶을 때

채워 넣을 입력:
- `[상위 목표]`

기대 결과:
- 작은 작업 단위 목록
- 순서
- 먼저 승인할 단위 추천

```text
다음 목표를 읽기 전용으로 작업 분해해 주세요: [상위 목표]. 구현은 하지 말고, 사람이 검토하고 승인하기 쉬운 작은 단위로 나누고, 우선순위와 의존 관계를 정리해 주세요. 첫 번째로 승인하기 가장 안전한 단위도 추천해 주세요.
```

## 5. 승인된 실행 프롬프트

언제 쓰는지:
- 이미 계획 결과를 보고, 아주 작은 범위를 승인했을 때

채워 넣을 입력:
- `[plan run id]`
- `[승인 메모]`

기대 결과:
- 승인된 범위만 실행
- worktree 기반 실행 보고

```text
아래 계획 결과를 이미 검토했고, 승인한 범위만 실행하고 싶습니다. plan run id: [plan run id]. 승인 메모: [승인 메모]. 승인한 범위를 벗어나지 말고, 실행 후 결과 아티팩트와 남은 위험을 요약해 주세요.
```

## 6. 리뷰 프롬프트

언제 쓰는지:
- 특정 실행 결과나 현재 상태를 읽기 전용으로 리뷰하고 싶을 때

채워 넣을 입력:
- `[run id 또는 대상 경로]`

기대 결과:
- 위험 요약
- 리뷰 결과
- 다음 추천 단계

```text
다음 대상을 읽기 전용으로 리뷰해 주세요: [run id 또는 대상 경로]. 문제점, 위험, 확인한 내용, 아직 확인하지 못한 내용, 추천 다음 단계를 정리하고 결과 아티팩트를 요약해 주세요.
```

## 7. 버그 수정 프롬프트

언제 쓰는지:
- 버그를 바로 고치기 전에 먼저 범위를 작게 계획하고 싶을 때

채워 넣을 입력:
- `[버그 설명]`
- `[재현 조건]`

기대 결과:
- 원인 가설
- 검증 계획
- 작은 수정 단위 제안

```text
다음 버그를 먼저 읽기 전용으로 분석해 주세요. 버그 설명: [버그 설명]. 재현 조건: [재현 조건]. 원인 가설, 확인할 로그 또는 증거, 가장 작은 수정 단위, 수정 후 리뷰 포인트를 제안해 주세요. 아직 구현은 하지 말아 주세요.
```

## 8. 문서 업데이트 프롬프트

언제 쓰는지:
- 코드보다 문서 정리나 가이드 업데이트가 먼저 필요할 때

채워 넣을 입력:
- `[문서 목표]`

기대 결과:
- 문서 변경 범위
- 업데이트 순서
- 리뷰 포인트

```text
다음 문서 작업을 읽기 전용으로 계획해 주세요: [문서 목표]. 어떤 문서를 바꿔야 하는지, 어떤 순서가 안전한지, 사용자가 헷갈릴 수 있는 부분이 무엇인지, 먼저 승인할 작은 문서 변경 단위를 정리해 주세요.
```

## 사용 팁

- 처음에는 첫 프로젝트 정의 프롬프트를 쓰는 것이 가장 안전합니다.
- 바로 구현을 시키고 싶더라도 먼저 계획 프롬프트를 한 번 거치는 편이 좋습니다.
- 실행 프롬프트는 반드시 이미 승인한 `plan run id`와 함께 써야 합니다.
